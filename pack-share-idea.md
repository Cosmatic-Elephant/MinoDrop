# 미노팩 공유 코드 시스템 — 아이디어 정리

## 현황 파악

`data-format.md`에 공유 코드 형식이 이미 명세로 정의되어 있다.

```
v{formatVersion}_{Base64URL(JSON.stringify(packData))}
예) v1_eyJuYW1lIjoi6Iqk66eI66...
```

그러나 현재 `library.js`의 `generateCode()`는 `crypto.getRandomValues`로 생성한
**랜덤 10자리 hex**를 `code` 필드로 쓰고 있어, 스펙과 구현이 불일치한 상태다.

즉 구현 목표는 **"새 기능 추가"가 아니라 스펙대로 완성하는 것**에 가깝다.

---

## 핵심 개념

```
[팩 객체] ──encode──> [공유 코드 문자열] ──붙여넣기──> [상대방 브라우저] ──decode──> [팩 객체]
```

- **공유 코드 = `code` 필드 그 자체**: 코드를 복사해서 붙여넣으면 팩 전체가 재구성된다.
- **별도 서버 불필요**: Base64URL이므로 텍스트만으로 오프라인 공유 가능.

---

## 인코딩 방식 비교

### 1안. Base64URL JSON (스펙 채택, 권장)

```
packData(code 필드 제외) → JSON.stringify → UTF-8 bytes → Base64URL → "v1_" 접두사
```

- 장점: 표준 브라우저 API(`btoa`, `TextEncoder`)만으로 구현, 디버깅 용이
- 단점: 코드 길이가 길다 (표준 7미노 팩 기준 약 **500~600자** 예상)
- 결론: 텍스트 박스에 붙여넣는 방식이라면 길이는 문제 없음

### 2안. 압축 Base64 (DecompressionStream)

```
JSON → CompressionStream('deflate-raw') → Base64URL
```

- 장점: 코드 길이 30~50% 단축
- 단점: `CompressionStream` API가 구형 브라우저 미지원, 비동기 처리 필요
- 결론: 지금 단계에서는 과하다. 1안으로 충분히 동작한 뒤 옵션으로 고려

### 3안. 커스텀 바이너리

- 비트 단위 직렬화로 가장 짧은 코드 생성
- 구현 복잡도가 너무 높고, 포맷 버전 관리 부담이 큼 → 보류

## 데이터 포함 범위

공유 코드에 인코딩되어야 할 필드:

| 필드 | 포함 여부 | 비고 |
|------|:---------:|------|
| `formatVersion` | ✓ | 역호환성에 필수 |
| `name` | ✓ | 팩 이름 |
| `thumbnail` | ✓ | library 카드 표시 |
| `minos[].name` | ✓ | |
| `minos[].color` | ✓ | |
| `minos[].size` | ✓ | |
| `minos[].shape` | ✓ | |
| `code` | ✗ | 코드 자체이므로 제외 후 재구성 |
| `size`, `minoCount` | △ | 파생 가능한 값 — 제외해도 import 시 재계산 가능 |

## 내보내기(Export) 흐름

```
[library.html] 팩 카드의 "코드 복사" 버튼 클릭
        ↓
encode(pack) 호출
  1. code 필드 제외한 팩 객체를 JSON.stringify
  2. TextEncoder로 UTF-8 바이트 변환
  3. btoa → URL-safe 치환 (+→-, /→_, = 제거)
  4. "v1_" 접두사 부착
        ↓
navigator.clipboard.writeText(code)
        ↓
토스트 알림: "코드가 클립보드에 복사됐습니다"
        ↓
(부수 효과) pack.code 필드를 새 코드로 업데이트 & savePacks() → 기존 랜덤 hex 교체
```

## 가져오기(Import) 흐름

```
[library.html] "코드로 불러오기" 버튼 클릭  ← 현재 없는 UI, 추가 필요
        ↓
모달 or 인라인 텍스트 입력창 표시
  (힌트 텍스트: "v1_... 형태의 공유 코드를 붙여넣으세요")
        ↓
사용자가 코드 붙여넣기 후 확인
        ↓
decode(codeString) 호출
  1. 접두사 확인 ("v1_" 없으면 오류)
  2. Base64URL 디코딩 → JSON 문자열
  3. JSON.parse → 팩 객체
  4. 유효성 검증 (아래 섹션 참조)
        ↓
중복 확인: 동일 code를 가진 팩이 이미 있는지 체크
  └ 있으면: "이미 추가된 팩입니다" 안내 (덮어쓰기 옵션 제공도 가능)
        ↓
pack.code = codeString (원본 코드 그대로 복원)
getPacks().push(pack) → savePacks()
        ↓
library 카드 목록 갱신
토스트 알림: "{팩 이름}을(를) 불러왔습니다"
```

## 유효성 검증 로직

decode 후 팩 객체 사용 전에 다음을 확인:

```
1. formatVersion이 지원 범위 내인가? (현재: 1)
2. name이 존재하고 32자 이내인가?
3. minos 배열이 존재하고 1~64개인가?
4. 각 mino에 대해:
   a. name이 존재하고 8자 이내인가?
   b. color가 /^[0-9a-f]{6}$/ 패턴인가?
   c. size가 2~20 범위의 정수인가?
   d. shape가 size×size 2D 배열인가?
   e. shape 셀 값이 모두 0 또는 1인가?
   f. shape에 채워진 칸이 최소 1개 이상인가?
```

검증 실패 시 → 어느 항목에서 실패했는지 사용자에게 표시

## 기존 팩 마이그레이션

현재 localStorage에 저장된 팩들의 `code`는 랜덤 hex 형식이다.
이를 새 Base64URL 형식으로 교체하는 방법:

**방식 A. 지연 마이그레이션 (Lazy)**
- 팩을 "처음 내보낼 때" 또는 "편집할 때" 새 코드로 교체
- 장점: 코드 단순, 기존 팩에 영향 없음
- 단점: 교체 전까지 import 불가

**방식 B. 일괄 마이그레이션**
- `packStore.js`의 초기화 시점에 구 형식(v1_ 없음)을 감지해 일괄 변환
- 장점: 모든 팩이 즉시 공유 가능
- 단점: 앱 로드 시 오버헤드, 코드 변경 시 다른 사람이 갖고 있던 기존 공유 코드 무효화

→ **A안 권장**: 공유 코드는 "내보낼 때" 생성하는 것이 자연스러운 흐름

---

## 코드 길이 예측 (참고)

표준 7미노 팩을 JSON 직렬화하면 약 **370~400 bytes**,
Base64 인코딩 시 `⌈N/3⌉ × 4` 공식으로 약 **500~540 chars** + `v1_` 접두사.

실제로는 미노 개수와 shape 크기(size)에 따라 선형으로 증가:
- 미노 1개 (3×3 shape) → 약 100~150자
- 미노 64개 (20×20 shape) → 약 20,000자 이상 → 클립보드 복사는 가능하나 URL 공유는 비현실적

## 구현 순서 제안

1. **`src/data/packStore.js`에 `encode(pack)` / `decode(codeStr)` 함수 추가**
   - 이 두 함수가 핵심. 나머지는 UI 연결에 불과함

2. **`library.js`의 "코드 복사" 버튼을 `encode()` 기반으로 교체**
   - 기존 랜덤 코드 → Base64URL 코드로 전환 + `pack.code` 갱신

3. **`library.html/js`에 "코드로 불러오기" 버튼 및 입력창 추가**

4. **유효성 검증 함수 작성**
   - `validatePack(packObj)` → `{ ok: true }` or `{ ok: false, reason: "..." }`

5. **중복 팩 처리 UX 결정** (덮어쓰기? 거부? 복사본 생성?)

## 미결 사항 / 추후 논의

- `createdAt` 필드를 공유 코드에 포함할지 여부
- `size`, `minoCount` 같은 파생 필드를 코드에 포함할지, import 시 재계산할지
- 코드가 너무 길어질 경우를 대비한 압축 옵션 (2단계 구현)
- 같은 팩을 공유받았을 때 중복 처리 UX
